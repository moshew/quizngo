# איפיון מערכת Dashboard לניהול תשתית משחקים (To-Be)

תאריך: 18 בפברואר 2026  
גרסה: 2.0 (Target Specification)

## 1. מטרה

המערכת תספק מרכז שליטה תפעולי לניהול אשכול שרתי המשחק (`srv`) דרך שרת ה-Load Balancer (`srv-lb`), עם תמונת מצב בזמן אמת ויכולת לבצע פעולות תפעול בטוחות.

הדשבורד יוגדר כמערכת ניהול תשתית, לא כמערכת ניהול תוכן משחק.

## 2. יעדים עסקיים

- לאפשר זיהוי מהיר של עומסים ותקלות בשרתי משחק.
- לאפשר ניתוב עומס מבוקר באמצעות `drain` ו-`activate`.
- לאפשר הסרה בטוחה של שרתים לא תקינים.
- לספק תחקור משחקים פעילים/ממתינים לפי שרת.
- להציג מדדי מערכת מאוחדים לצורך קבלת החלטות תפעוליות.

## 3. גבולות והיקף

### In Scope

- תצוגת מצב שרתים, PINs ומדדי עומס.
- פעולות ניהול שרתים דרך API של `srv-lb`.
- תצוגת משחקים לשרת ספציפי.
- רענון ידני ואוטומטי של נתונים.
- הצגת מגמות קצרות טווח (Trend) למדדים מרכזיים.

### Out of Scope

- ניהול שאלות, שקפים, תוכן טריוויה וניקוד.
- ניהול משתמשי קצה של המשחק.
- החלפת מנגנון הבחירה של שרתים ב-`srv-lb`.

## 4. שחקנים

- מפעיל תפעול (Operator): צופה במדדים ומבצע פעולות `drain`/`activate`/`remove`.
- מערכת `srv-lb`: מקור האמת לסטטוס שרתים ולמיפוי PIN.
- שרתי `srv`: מקור המידע למשחקים מקומיים וסטטיסטיקות heartbeat.

## 5. ישויות נתונים

### Server

- `server_id: string`
- `address: string`
- `registered_at: unix_seconds`
- `last_heartbeat: unix_seconds`
- `status: "active" | "draining" | "down" | "unknown"`
- `stats.active_ws_connections: number`
- `stats.cpu_percent: number`
- `stats.memory_mb: number`
- `stats.active_games_count: number`
- `active_pins: string[]`

### PIN Mapping

- `game_pin: string`
- `server_id: string`
- `server_address: string`
- `assigned_at: unix_seconds`

### Server Game

- `gamePin: string`
- `timestamp: unix_seconds`
- `active: boolean`
- `gameStarted: boolean`

## 6. דרישות פונקציונליות

### FR-01: הגדרת כתובת LB

- המערכת תתמוך בכתובת LB ברירת מחדל מתוך `VITE_LB_URL`.
- אם המשתנה אינו מוגדר, המערכת תשתמש ב-`http://<hostname>:5000`.
- המערכת תאפשר שינוי כתובת LB בזמן ריצה.
- המערכת תנרמל URL (הסרת `/` מסיים) לפני כל קריאה.
- המערכת תחסום הפעלה עם כתובת לא תקינה ותציג שגיאה ברורה.

### FR-02: טעינת נתונים ורענון

- בטעינה ראשונית המערכת תטען במקביל:
- `GET /api/admin/servers`
- `GET /api/admin/pins`
- המערכת תתמוך ברענון ידני.
- המערכת תתמוך ברענון אוטומטי בתדירויות: `כבוי`, `5s`, `10s`, `30s`.
- המערכת תציג זמן עדכון אחרון.
- בזמן שגיאה, המערכת תציג הודעת שגיאה למפעיל ולא תקרוס.

### FR-03: חישובי KPI מערכתיים

- המערכת תחשב ותציג:
- סך שרתים.
- כמות שרתים לפי סטטוס (`active`, `draining`, `down`).
- סך חיבורי WS.
- סך משחקים פעילים.
- סך PINs פעילים.
- סך שימוש זיכרון (MB).
- ממוצע CPU לכלל השרתים.

### FR-04: מגמות מדדים

- המערכת תשמור היסטוריית דגימות מדדים לחלון נע של 30 נקודות.
- המערכת תציג גרפים עבור:
- ממוצע CPU.
- סך זיכרון.
- סך חיבורי WS.
- סך משחקים.
- כל דגימה תילקח בכל רענון מוצלח.

### FR-05: תצוגות שרתים

- המערכת תספק לפחות שתי תצוגות: `table` ו-`cards`.
- שתי התצוגות יציגו אותם נתוני ליבה לכל שרת.
- שתי התצוגות יתמכו באותן פעולות ניהול שרת (כולל `remove`).
- המערכת תמיין שרתים בצורה עקבית (סטטוס ואז `server_id`).

### FR-06: פעולות ניהול שרת

- פעולת `drain` תהיה זמינה רק לשרת בסטטוס `active`.
- פעולת `activate` תהיה זמינה לשרת שאינו `active`.
- פעולת `remove` תדרוש אישור משתמש ותבצע הסרה מלאה של השרת והמיפויים.
- המערכת תנעל כפתור פעולה בזמן בקשה בתהליך עבור אותו שרת.
- לאחר פעולה מוצלחת המערכת תרענן נתונים אוטומטית.
- המערכת תציג הודעת הצלחה או כשלון לכל פעולה.

### FR-07: פאנל משחקים לשרת

- המערכת תאפשר פתיחת פאנל משחקים מתוך שורת שרת.
- המערכת תציג מצבי `loading`, `error`, `empty`, ו-`data`.
- בתצוגת נתונים יוצגו לכל משחק:
- `gamePin`
- `active`
- `gameStarted`
- `timestamp`
- המערכת תאפשר סגירת פאנל וחזרה למסך הראשי ללא רענון מלא.

### FR-08: חוזה שגיאות בצד לקוח

- כל קריאת API תעבור דרך שכבת Client אחידה.
- אם `HTTP status` אינו `2xx`, הקריאה תיחשב ככשלון.
- אם גוף תשובה מחזיר `status="error"`, הקריאה תיחשב ככשלון גם ב-`2xx`.
- הודעת שגיאה תיגזר לפי סדר עדיפות: `message` ואז `error` ואז fallback גנרי.

### FR-09: תאימות לכללי דומיין של LB

- המערכת תיישר קו עם הגדרות סטטוס:
- `active`: מקבל משחקים חדשים.
- `draining`: לא מקבל משחקים חדשים, ממשיך לשרת קיימים.
- `down`: לא כשיר לשירות.
- המערכת תציג heartbeat יחסית לזמן נוכחי.
- המערכת תציג PINs פעילים לכל שרת על בסיס נתוני LB.

### FR-10: התממשקות לנתוני משחקים

- המערכת תצרוך רשימת משחקים בפורמט `games[]` כפי שמוחזר מ-`/sim_gamePIN`.
- אם מקור הנתונים אינו זמין, המערכת תציג שגיאה ממוקדת לשרת הרלוונטי בלבד.

## 7. חוזה API נדרש

### API Load Balancer (Admin)

- `GET /api/admin/servers`
- `GET /api/admin/pins`
- `POST /api/admin/servers/{server_id}/drain`
- `POST /api/admin/servers/{server_id}/activate`
- `DELETE /api/admin/servers/{server_id}`

### API שרת משחק (חקירה)

- `GET {server_address}/sim_gamePIN`

### מבנה תגובת הצלחה אחיד

- `status: "success"`
- payload ייעודי ל-endpoint

### מבנה תגובת שגיאה אחיד

- `status: "error"`
- `message: string`

## 8. דרישות לא פונקציונליות

### NFR-01: ביצועים

- זמן עדכון dashboard לרשימת שרתים ופינים: עד 2 שניות ב-LAN תקין.
- פעולת ניהול שרת (`drain`/`activate`/`remove`) תושלם ותשוקף ב-UI בתוך עד 3 שניות במצב תקין.

### NFR-02: אמינות

- המערכת תמשיך לתפקד חלקית גם אם קריאה בודדת נכשלת.
- כשל בקריאת משחקים לשרת אחד לא יפגע בתצוגת כלל השרתים.

### NFR-03: סקייל

- המערכת תתמוך בלפחות 200 שרתים ו-10,000 PIN mappings ללא קיפאון UI מורגש.

### NFR-04: שימושיות

- כיוון ממשק: `RTL`.
- כל הטקסטים למשתמש יהיו באותה שפה (עברית מלאה או אנגלית מלאה לפי קונפיגורציה).
- המסך יהיה רספונסיבי בדסקטופ ובמובייל.

### NFR-05: נגישות בסיסית

- כל פעולה תהיה נגישה דרך מקלדת.
- לכל כפתור יהיה תיאור ברור.
- ניגודיות צבעים תעמוד בתקן AA לפחות.

## 9. אבטחה והרשאות

- כל endpoint תחת `/api/admin/*` יחייב אימות משתמש.
- פעולות ניהול שרת יחייבו הרשאה מסוג `ops-admin`.
- המערכת תתעד Audit Log עבור פעולות `drain`/`activate`/`remove`.
- CORS יוגבל לרשימת מקורות מאושרים בלבד בסביבות Production.

## 10. כללי תפעול מחייבים (System Semantics)

- Heartbeat מהשרתים יישלח כל 30 שניות.
- LB יסמן שרת כ-`down` לאחר 90 שניות ללא heartbeat.
- בדיקת בריאות תרוץ כל 60 שניות.
- ניקוי PINs לא עדכניים ירוץ כל 600 שניות באמצעות polling ל-`/sim_gamePIN`.
- בחירת שרת למשחק חדש תתבצע רק מתוך שרתים `active` ובריאים.
- מיון לבחירה: `active_games_count` ואז `active_ws_connections`.

## 11. קריטריוני קבלה

### AC-01: רענון אוטומטי

- כאשר מוגדר `10s`, הנתונים יתעדכנו כל 10 שניות ללא רענון דף.

### AC-02: מעבר ל-drain

- לחיצה על `drain` בשרת `active` תשנה סטטוס ל-`draining` לאחר רענון.

### AC-03: הפעלה מחדש

- לחיצה על `activate` בשרת `draining/down` תחזיר אותו ל-`active` (אם השרת זמין ב-LB).

### AC-04: הסרה

- `remove` לאחר אישור תסיר את השרת מרשימת השרתים ותנקה PIN mappings משויכים.

### AC-05: פאנל משחקים

- פתיחת פאנל משחקים תציג רשימה נכונה של משחקים או הודעת ריק/שגיאה בהתאם לתשובה.

### AC-06: שגיאות API

- כשל API יוצג כהודעת שגיאה קריאה למשתמש ללא שבירת שאר המסך.
